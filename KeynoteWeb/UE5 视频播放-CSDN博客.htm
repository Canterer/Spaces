关键字：MediaSource、MediaPlayer、MediaTexture、electra player插件、MediaSoundComponet、OpenSource函数

MediaFramework媒体框架与MediaSource媒体源：

        MediaSource为基类，针对不同的文件类型区分为不同派生类。如UPlatformMediaSource、UStreamMediaSource、UFileMediaSource、UImgMediaSource等。其中UMediaSource继承于IMediaOptions，IMediaOptions提供虚函数GetDesiredPlayerName给派生类返回文件名，其也是转换媒体解码模块的关键判断。核心类为MediaPlayer类型为UMediaPlayer，其中成员变量FMediaPlayerFacade* PlayerFacade负责封装播放媒体的框架。

UMediaPlayer负责整体对外，FMediaPlayerFacade负责封装媒体播放的各种细节。UMediaPlaylist管理播放列表。FMediaPlayerFacade持有成员变量IMediaModule* MediaModule、IMediaPlayer* Player。

其中IMediaModule为媒体总模块，用于管理加载各种媒体插件模块，比如ElectraPlayerFactoryModule、FWmfMediaModule等。其中各种媒体通过IMediaModule::RegisterPlayerFactory注册各种媒体模块的MediaPlayerFactory工厂类，后者IMediaPlayerFactory提供创建接口virtual TSharedPtr<IMediaPlayer, ESPMode::ThreadSafe> CreatePlayer(IMediaEventSink& EventSink)返回一个具体支持IMediaPlayer的派生类用于播放。

        

总结：FMediaPlayerFacade持有IMediaModule，后者持有所有的媒体插件注册的MediaPlayerFactory工厂类，知晓所有支持的媒体播放形式。第一次播放时通过支持播放源的工厂类创建IMediaPlayer 调用虚函数Open播放。

UMediaTexture创建材质球，赋予UI元素或Panel。其中UMediaTexture需要关联MediaPlayer，UMediaTexture::TickResource其中会通过MediaPlayer获取是否播放及播放的速度与时间，来确定显示当前画面帧。

MediaSoundComponent继承于USynthComponent，继承来的SetVolumeMultiplier用于调节声音大小。其关联MediaPlayer，UMediaSoundComponent::TickComponent中调用UpdatePlayer一样为获取播放的时间段以及音频采样数据，OnGenerateAudio()基于这些数据用于输出音频数据。

针对Android声音控制：FileMediaSource可以指定在不同平台采用不同的解码器。最初创建项目时，针对MP4媒体Windows默认为WMF解码，这时看不到画面。安装ElectraPlayer插件后，Windows下播放MP4媒体会有画面(仍无画面手动设置Windows采用Electra)。在UE5版本改动了音频系统底层。为了播放声音，需要使用Actor负载MediaSoundComponent并为其指定MediaPlayer(主要时为了获取播放进度等)。当需要控制其音量大小时，在Windows下需要使用MediaSoundComponent->SetVolumeMultiplier控制，在Android下需要使用MediaPlayer->SetNativeVolume控制。项目开发应该针对不同播放平台手动设置辅助控制类。

针对Seek定位不准：Android平台的MediaPlayer:seekTo函数有两个重载方法，其中一个会多传入一个mode枚举用于控制seek。Android的seek会默认跳转到给定时间附近的关键帧，导致seek不准。SEEK_CLOSEST模式会跳转最近的帧而不是关键帧，其中UE的androidMediaPlayer不提供额外参数的传入，导致需要使用ffmpg对源视频进行关键帧间隔调整。命令：ffmpeg.exe -i "F:\1.mp4" -c:v libx264 -preset superfast -x264opts keyint=5 -acodec copy -f mp4 "F:\2.mp4"其中keyint=5代表每5帧设置一个关键帧。另外一种解决方案：在AndroidMediaPlayer依赖的FJavaAndroidMediaPlayer::GetClassName()获取一个文件名如FName("com/epicgames/unreal/MediaPlayer14")，UE引擎的安装目录可以找到对应的文件(在build文件夹下)，修改其中的Seek方法，在其中传递SeekMode枚举值。修改后使用Android Studio打开对应的工程build一下。例如C:\Program Files\Epic Games\UE_5.0\Engine\Build\Android\Java\src\com\epicgames\unreal\MediaPlayer14.java路径，其中对应的模块为Android\Java，Android Studio打开工程选择该目录，然后build->Make Module Java。最后在项目里打包出的apk运行seek播放定位就能接受。其中UE里的音频播放会有一点重复，其中需要修改为模式SEEK_CLOSEST_SYNC。音频与视频就能同步播放无异常。查看seek mode枚举值网址：MediaPlayer  |  Android Developers

针对大视频URL卡顿：

环境：需要部署远程文件存储系统，上传文件然后获取其URL。一般采用各个大公司提供的云存储系统。本地测试环境：python3.8部署轻量本地文件服务器，命令：python -m http.server port 本机测试游览器输入http://127.0.0.1:port/filename下载文件，其中IP地址可查询本机提供给远程下载。端口默认为8000，命令行输入netstat -ano或netstat -ano | findstr "8080"查看端口使用情况。

原始文件如mp4在UE中通过URL点播时，其中自适应码率在卡顿时会反复切换码率导致来回重建buffer。在不能降低视频的码率(画面质量)情况下，原始mp4文件不太能接受URL形式。在线电影播放一般都是采用HLS基于HTTP的自适应码率流媒体传输协议，其中通过m3u8+ts来实现点播的。

一般在线电影的m3u8不能直接被UE的ElectraPlayer接受。https://docs.unrealengine.com/5.0/zh-CN/electra-media-player-in-unreal-engine/针对播放列表和媒体片段具有限制。需要ffmpeg针对mp4进行分片转换，配合相关参数生成能被接受的m3u8文件。【译】如何制作虚幻Electra Media Player的HLS流媒体文件 - 知乎

./ffmpeg.exe -i 原始视频.mp4  -c:v copy -c:a copy 
-hls_list_size 0 -hls_segment_type fmp4 -f hls 
-master_pl_name 主播放列表.m3u8 -hls_fmp4_init_resend 1 
-var_stream_map "v:0,agroup:audio_group a:0,agroup:audio_group,language:JP"
 次播放列表_%v_.m3u8

其中./ffmpeg.exe为未配置其环境变量方式。-c:v copy为设置视频编码 copy指明采用源视频方式，-c:a代表音频。常用编码格式为-c:v h264 -c:a aac，默认每个m3u8的分片采用最后5个，-hls_list_size 0代表记录所有。-hls_segment_type fmp4指明分片格式，其中ElectraPlayer仅支持m4s分片文件(不支持ts)。-master_pl_name主导m3u8带关键字#EXT-X-STREAM-INF，这种形式才符合ElectraPlayer。

扩展知识：

每个输入或输出文件，原则上都可以包含任意数量的流。FFmpeg 中流的类型有五种：视频(video)、音频(audio)、字幕(subtitle)、附加数据(attachment)、普通数据(data)。文件中流的数量和(或)流类型种数的极限值由文件封装格式决定。选择哪一路输入文件的哪一路流输出到哪一路输出，这个选择过程既可以由 FFmpeg 自动完成，也可以通过 “-map” 选项手动指定(后续 “Stream selection” 章节会深入描述)。

// -c:v -c:a代表视频编码 音频编码  copy代表采用输入文件编码

// req_%v_.m3u8 自动在名字后面加切片的序号    %v对应输出m3u8的序号 可生成多个次级m3u8文件

 ./ffmpeg.exe -i report1_1.mp4  -c:v copy -c:a copy -hls_list_size 0 -hls_segment_type fmp4 -master_pl_name rep.m3u8 -var_stream_map "v:0,agroup:1_group a:0,agroup:1_group" C:/Users/Administrator/Desktop/hls/req_%v_.m3u8
// -master_pl_name 主m3u8文件名
// -var_stream_map 指明输入输出流的映射关系 空格区分生成不同的次级m3u8文件 
// v:0指明输出流类型为视频的 数组第一个输出  可存储多个视频输出内容 分别指明映射关系。
// 一个媒体文件 可存储多个不同类型的流数据数组 如视频类型、音频类型、字幕类型等
// 每个类型对应一个数组 如两个视频流数据  其中一个用于水印overlay另一个 即覆盖在另一个画面上
// 流数据数组对应代码的track轨道  track分不同的类型视频、音频等
// 多个音频数据流 可以在一个输出设备如扩音器播放  每个音频数据流即track轨道都可设置声音 最好混合所以数据给输出设备  所以对音频数据设置声音API有类似的setTrackVolume设置某一个流轨道的音量

只有主次级m3u8播放列表模式下(-var_stream_map非必须) m3u8才会显示CODECS BANDWIDTH RESOLUTION
agroup:xx 主要用于指明音频所属的组 用于与视频m3u8播放列表绑定   两者的数据会分开一起下载使用
BANDWIDTH 这里显示是原文件码率 对应文件属性详情中视频音频的比特率 


-b:v 1M 用于限制码率 即修改BANDWIDTH  主要用于动态适应不同网速 选择不同码率
将同一份数据分割为不同码率的播放列表 播放时根据网速来选择不同码率的分片下载播放
仅使用-b:v 只是影响配置数据 影响播放选项 如buffer缓存 动态码率适应等 理论上会改变文件大小 实际需要搭配其他参数才会影响
-b:v 1M -maxrate:v 1M -minrate:v 1M -bufsize:v 2M 相关播放参数设置

//-filter_complex为复杂滤镜 具体细节上面链接可参考
//  "[0:v]split=2[v1][v2];[v1]copy[v1out];[v2]scale=w=640:h=360[v2out]"
// 0:v代表第一个输入文件的视频流数组 源视频默认仅一个视频流 就没有使用 0:v:0
// [v1][v1out]为中间变量 重点关注map
// -map [v1out] 使其成为后面命令输入的 v:0    源视频是没有v:1的  其中-var_stream_map的v:1指代第二个-map [v2out]
// "v:0,agroup:720_group a:0,agroup:720_group v:1,agroup:360_group a:1,agroup:360_group"
// 其中视频被分割(可理解为拷贝)成两份 其中一份在-filter_complex中修改了分辨率 两份画面搭配需分别搭配同一份音频数据  仅一个-map a:0则造成Same elementary stream found more than once in two different variant definitions


注:为什么将v:0 a:0 分成两个m3u8  其中原因为UE的ElectraPlayer仅支持m4s格式的分片，同时分片仅允许一个track,若音频和视频合并会崩溃Assertion failed: MP4Parser->GetNumberOfTracks() == 1
