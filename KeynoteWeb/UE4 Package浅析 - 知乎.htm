前言

之前阅读UE反射系统源 码的时候，里面有一个Package的概念。在注册的步骤会创建一个Package对象作为UClass对象的OuterPrivate(也就是通常所说 的Outer对象)。那时候看了一些资料脑海里面有一个模糊的概念，对象通过Outer来定义从属关系，也就是说该UClass对象从属于这个包。于是， 想更进一步了解Package的概念，所以就找到了虚幻官方的这个视频。得知一个UE中Package的概念是一个文件，用来表示各种类型的资产(.uasset、.umap等等)。

从新建蓝图资产开始

既然一个Package在UE中表示一个资产，那我们新建一个Actor蓝图看看会发生什么？

当我们新建一个Actor蓝图时，会调用到CreatePackage()。我们可以看到包名是从一个AssetData结构体里面得到的：

FAssetData

FAssetData是一个用来保存资产信息的结构体。

可以看到PackagePath就是我们新建蓝图的文件夹路径。而新建的蓝图资产(完整的包名)就是PackagePath下的AssetName(NewBlueprint1)，也就是PackagePath/AssetName。

该函数很简单，传入了一个完整的包名，先判断该Package对象是否存在，如果存在就使用该对象，否则调用NewObject<>()来创建Package对象(注意它传入的Outer为nullptr，那在函数里面会将GObjTransientPkg作为它的Outer对象)。NewObject流程这篇文章已经详细分析，这里就不重复讲解了。

好 了，现在Package对象已经创建出来，那按照之前了解的知识，创建一个对象必须传入一个对象(可以是普通的Object对象也可以是Package对 象)作为其Outer。我们创建一个蓝图必定会创建一些对象，那创建的这些对象会将这个Package对象作为它们的Outer吗？我想大概率会，为了验 证我的猜想，可以在StaticConstructObject_Internal()函数内部打一个断点，该函数是NewObject内部调用的函数。

果然，会创建一个UBlueprint对象，并且会将刚刚创建出来的Package对象作为它的Outer对象。

当调用NewObject<>()来创建一个对象，最终会调用到UObjectBase::AddObject()函数，将该对象添加到全局Object数组和各种hash表中，关于这两个我在这篇文章已有详细分析，这里也不详细分析了。

这里看一下添加到hash表的流程，因为这个后面需要用到。

AddObject()函数里面会调用一个全局函数HashObject()，该函数就是将该对象添加到各种hash表的关键。

我们可以明显的看到将该对象添加到了Hash、HashOuter、ObjectOuterMap、ClassToObjectListMap这四个Map里面，而值得注意的是HashOuter的key是该对象的Outer转换而成的，请记好这个关键点，下文将会要用到。其它和该蓝图相关的对象也会一一添加到这几个hash表中。

从上文分析可知，当我们创建一个蓝图资产时，会随之创建一个Package对象，该Package名为Game/Path/AssetName。但是我们平时在C++开发时，需要将一个蓝图动态创建出来，那要怎么做呢？

在代码中动态创建蓝图对象

通 常在游戏开发中，我们可能会有很多角色，通常的做法就是在C++中定义一个基类，然后在UE中创建子类角色蓝图，它们的mesh、动画蓝图等等可能会不 同。那现在有一个需求，进入游戏创建一个新号的时候需要区分性别，那角色蓝图就必定需要动态生成。那我们该怎么将它动态生成出来呢？

通常可以调用LoadClass<>()来加载UClass对象，然后通过这个UClass对象将角色蓝图Spawn出来。

UClass* CharacterClass = LoadClass<AYJHCharacter>(this, TEXT("Blueprint'/Game/ThirdPersonCPP/Blueprints/ThirdPersonCharacter.ThirdPersonCharacter_C'"));
AActor* Character = UGameplayStatics::BeginSpawningActorFromClass(GetWorld(), CharacterClass, FTransform(FRotator::ZeroRotator, FVector(-700.f, 370.f, 226.f), FVector::OneVector));

新建一个第三人称模板工程，通过上述代码就可以正确的在运行时创建角色蓝图。

好像看到了熟悉的东西，/Game/ThirdPersonCPP/Blueprints/ThirdPersonCharacter这个不就是ThirdPersonCharacter这个蓝图资产的包名嘛。那前面的"Blueprint"和后面的"ThirdPersonCharacter_C"代表的是什么意思呢？让我们带着疑问看LoadClass<>()调用的时候里面发生了什么。

其核心是调用StaticLoadObjectInternal()来加载一个对象。

{

    UObject* Result = nullptr;
    const bool bContainsObjectName = !!FCString::Strstr(InName, TEXT("."));

    if (InOuter)
    {

        if (bAllowObjectReconciliation && (bContainsObjectName))
        {
            Result = StaticFindObjectFast(ObjectClass, InOuter, *StrName);
            if (Result && Result->HasAnyFlags(RF_NeedLoad | RF_NeedPostLoad | RF_NeedPostLoadSubobjects | RF_WillBeLoaded))
            {

                Result = nullptr;
            }

该函数主要这几件事：

  • 调用ResolveName()函数，处理我们传入的Outer对象和Name。将我们传入的Outer对象改成了ThirdPersonCharacter这个蓝图资产对应的Package；将传入的Name中的"ThirdPersonCharacter_C"提取出来。而前面的"Blueprint"也被去掉了，也就是说"Blueprint"没什么用。
  • 调用StaticFindObjectFast()函数来find该对象，其本质是调用StaticFindObjectFastInternalThreadSafe()函数。

StaticFindObjectFastInternalThreadSafe()函 数里面会通过传入的StrName和InOuter来获取一个Hash，然后在FUObjectHashTables里面的HashOuter里面寻找我 们需要的对象。HashOuter是不是感到很熟悉？Ctrl + F，跳转回顾上文。上文说到，当我们创建一个蓝图资产的同时，会创建一个Package对象，也会创建其它很多的对象，它们会将该Package对象作为 自己的Outer对象，最后会将Package对象转换为key，创建出来的对象为value映射到FUObjectHashTables里面的 HashOuter中。注意HashOuter是一个TMultiMap，它允许存在相同的key，也就允许同一个Package对象对应多个其它对象。

我们可以看到找到的对象是一个名为"ThirdPersonCharacter_C"的UBlueprintGeneratedClass对象。UBlueprintGeneratedClass表示蓝图编译生成的类，它继承自UClass，所以我们也可以通过该UBlueprintGeneratedClass对象来生成我们的角色蓝图。

现在终于明白了在LoadClass<>()函数里面填入的字符串里面的"ThirdPersonCharacter_C"代 表的是伴随该蓝图资产生成的UBlueprintGeneratedClass对象名。在创建蓝图资产时，会伴随着生成一个 UBlueprintGeneratedClass对象，它继承自UClass，表示蓝图编译生成的类。而我们需要在代码中动态创建一个蓝图，就需要用到 这个UBlueprintGeneratedClass对象，而这个对象需要通过"PackageName.YouBuleprintName_C"来获取。

总结

这 篇文章分析了Package的作用，UE通过对象的Outer来表示对象的从属关系，通常对象的Outer是一个Package对象，也可以是一个普通的 对象，但最外层的Outer对象一定是一个Package对象。并且会在创建对象的时候将它们映射到hash表中，所以我们可以通过一个Package对 象可以获取Package对象映射的其它的对象。所以也可以说是这些对象从属于这个包中。

那这样就能解释一个Package在UE中的概念是一个文件了嘛？我觉得不行，我觉得还需要加上序列化和反序列化。以下是我个人的猜测：

我觉得UE会通过Package来获取它包含的所有对象，然后将这些对象一起序列化到文件中，然后打开Editor的时候会将文件反序列化到内存里面，所有才有了编辑器里面现实的资产。为了验证我的猜想，接下来研究UE的序列化！
