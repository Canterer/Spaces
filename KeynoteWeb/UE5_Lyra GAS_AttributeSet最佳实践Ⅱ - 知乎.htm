Executions

GameplayEffectExecutionCalculation(ExecutionCalculation, Execution(你会在插件代码里经常看到这个词)或ExecCalc)是GameplayEffect对ASC进行修改最强有力的方式. 像ModifierMagnitudeCalculation一样, 它也可以捕获Attribute并选择性地为其创建Snapshot, 和MMC不同的是, 它可以修改多个Attribute并且基本上可以处理程序员想要做的任何事. 这种强有力和灵活性的负面就是它是不可预测的（Modifiers是可以预测的）且必须在C++中实现.


ExecutionCalculation只能由即刻(Instant)和周期性(Periodic)GameplayEffect使用, 插件中所有和"Execute"相关的一般都引用到这两种类型的GameplayEffect.


当 GameplayEffectSpec创建时, Snapshot会捕获Attribute, 而当GameplayEffectSpec应用时, 非Snapshot会捕获Attribute. 捕获Attribute会自ASC现有的Modifier重新计算它们的CurrentValue, 该重新计算不会执行AbilitySet中的PreAttributeChange(), 因此所有的限制操作(Clamp)必须在这里重新处理.

对于Local Predicted, Server Only和Server Initiated的GameplayAbility, ExecCalc只在服务端调用.


ExecCalc最普遍的应用场景是计算一个来自很多源(Source)和目标(Target)中Attribute伤害值的复杂公式. 样例项目中有一个简单的ExecCalc用于计算伤害值, 其从GameplayEffectSpec的SetByCaller中读取伤害值, 之后基于从目标(Target)捕获的护盾Attribute来减少该伤害值. 参阅GDDamageExecCalculation.cpp/.h.


Lyra 将其分为两类：Heal只管Buff，Damage只管Debuff


Attribute的永久值和最终值

BaseValue永久值（the value of the attribute with no stateful modifiers），可以对局状态内，持久化存储
CurrentValue最终值


一 个Attribute是由两个值组成， 一个BaseValue和一个CurrentValue组成的, BaseValue是Attribute的永久值而CurrentValue是BaseValue加上GameplayEffect给的stateful Modifiers后得到的. 例如, 你的Character可能有一个BaseValue为600u/s的移动速度Attribute, 因为还没有GameplayEffect修改移动速度, 所以CurrentValue也是600u/s, 如果Character获得了一个stateful Modifiers——50u/s的移动速度加成, 那么BaseValue仍然是600u/s而CurrentValue是600+50=650u/s, 当该移动速度加成消失后, CurrentValue就会变回BaseValue的600u/s.


有多种方法可以初始化Attribute(将BaseValue和CurrentValue设置为某初始值). Epic建议使用即刻(Instant)GameplayEffect, 这也是样例项目使用的方法.


只要使用了GE，都会使用stateful Modifiers

stateful Modifiers的最终目的是叠加到BaseValue之上，修改BaseValue；

总结

HealthSet 会同时负责 Atrribute 和 Meta Attribute


CombatSet从状态上来说，BaseDamage, BaseHeal 属于Attribute ，但是从作用性上说类似 Meta Attribute（Meta Attribute不会被强制Clamp，且不会被复制回客户端，不会在对局状态内，持久化存储）

BaseDamage, BaseHea虽然不会强制Clamp（本质在GE Executions配置参数即可），但是确实是会复制回客户端的，且只会发送到 Actor's owner，因为针对施加方，施加类Attribute只需要Actor自己知道就可以，BaseDamage会随着GE Executions 广播到所有客户端 Target，在此之前一局游戏里，其他Actor理应是一无所知的。


与之对应，针对接收方，接受类Attribute，Health，MaxHealth 需要发送到所有 Actor，因为一旦受到影响，就立即需要在所有端口，所有 Actor 之间同步Attribute，逻辑层和表现层也是。


逻辑层交给Executions就可以，而表现层还需要其他的协助，HealthComp的配合，与UI交互、动画反馈交给Gameplay Cue，多端同步播放受击反馈蒙太奇等等
